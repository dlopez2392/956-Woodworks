1. System Overview

956woodworks.com is a fully static marketing site for a custom woodworking business in the Rio Grande Valley. There is no backend server, no database, and no build step: five self-contained HTML pages plus a 404 page are served directly from Vercel's CDN. Interactivity (galleries, the commission form, the photo lightbox) runs entirely in the visitor's browser; the only write path out of the site is the contact form, which posts to the third-party Web3Forms API and arrives as email.

The pages were designed in a visual editor ("DesignCanvas" / DC) whose export format shapes the whole architecture: each exported page embeds its own copy of the DC runtime, page markup, and component logic as gzip-compressed base64 blobs inside the HTML, hydrated client-side with React 18. Because that bundle cannot be edited directly, all post-export fixes and features are delivered as small, network-loaded runtime scripts (section 7).

Anatomy of an exported page

Each site/*.html file contains, in order: a hand-maintained static <head> (SEO meta, JSON-LD, preload hints, an inline fetch-redirect script, a head-restore script); the DC bundle (React loader + <script type="__bundler/manifest"> blobs holding the compressed runtime, markup, and component logic); and the runtime fix scripts plus analytics tags before </body>. At load time the DC engine decompresses its blobs, mounts the page with React, and replaces the entire <head> and <html> attributes — which is why an inline capture-and-restore script re-applies the SEO tags and a11y.js re-asserts lang="en".

4. The Image-Slot Photo System (core subsystem)

All 87 photos live as base64 data-URLs inside JSON "sidecar" files, not as image files. Each <image-slot id="..."> element fetches the sidecar, looks up its id, and renders the photo in shadow DOM with cover-fit and reframe values ({u, s, x, y} = image, scale, x/y offsets in frame-%). The bundled loader rejects any non-data:image/ URL, so photos must stay base64. Slot ids are permanent — reusing or renumbering one silently swaps photos (this caused real incidents; rule is codified in CLAUDE.md).

For performance, the editor's single master file is split per page and recompressed by tools/sync-image-state.js. An inline <head> script wraps window.fetch to redirect the runtime's request for the full file to the page-specific one (falling back to the full file if missing) and caches the response, because the bundle evaluates the loader module twice on the gallery page.

Performance history. The original export shipped one 5.7 MB sidecar to every page (the gallery downloaded it twice = 11.4 MB). After converting 21 JPEG/PNG photos to WebP q80, downscaling 23 oversized photos to 660 px, splitting per page, deduplicating the double-fetch, and preloading: the home page now transfers ~640 KB and the gallery ~1.7 MB (Brotli-compressed on the wire). Lighthouse (mobile): Accessibility 100, SEO 100, Best Practices 96, Performance 42 — the last is bound by the DC client-side rendering architecture, not asset weight.

7. Runtime Fix Scripts (the "patch layer")

Because the DC export embeds its logic in compressed blobs, the live site is enhanced by small network-loaded scripts. Convention: every fix also patches the DC source files where possible so a future re-export bakes it in; injected DOM carries ids and is re-applied by slow maintenance ticks; event handlers are delegated to document (the DC engine re-renders every few seconds and drops per-node listeners, but keeps injected nodes).

9. Maintenance Runbook

Golden rule №1 — slot ids are permanent. Never renumber, reorder, or reuse an <image-slot> id. New photos get brand-new ids (bump a counter: gal-226, gal-227…). Ids are the only key into the photo store.
Golden rule №2 — after ANY image change, run node tools/sync-image-state.js. The live pages read the per-page files; the full-file fallback only covers a missing page file, not a stale one. Skipping this shows old photos on the live site.

Common tasks

  • Deploy anything: commit and push to main. Live in ~30–60s. Verify at 956woodworks.com.
  • Add/replace a photo: edit in the DC editor (writes the root state file) → run the sync tool → push.
  • Change copy/layout: edit the DC source, re-export, then work the re-export checklist below.
  • Rollback: git revert the commit and push (or promote an older deployment in the Vercel dashboard).

Re-export checklist (whenever site/ is regenerated from the DC editor)

  • Copy image-slot.js into site/ (the export drops it).
  • Fix the 5 nav links in site/index.html (.dc.html names → kebab-case pages) — or rely on the vercel.json redirects.
  • Re-add the defer script tags for the 7 runtime scripts + the two Vercel analytics tags before </body>.
  • Confirm the inline head scripts (SEO restore, fetch-wrapper, preload link) survived — they live in the source helmets, but verify.
  • Run node tools/sync-image-state.js; keep 404.html, vercel.json, sitemap/robots/icons in place.

Known limitations & watch items

  • Mobile Lighthouse performance (~42) is bound by DC's client-side full-page rendering (~6s main-thread on a throttled phone). Fixing it means re-platforming; real-world fast connections are fine (CLS = 0).
  • Web Analytics toggle pending in the Vercel dashboard (script route 404s until enabled + redeployed); this is also the only remaining Lighthouse Best-Practices deduction (console 404).
  • Testimonials are anonymous placeholders — swap in real quotes (name + city) when available; never fabricate.
  • Third-party dependencies at runtime: unpkg (React) and Google Fonts. If unpkg is ever slow/down, pages degrade badly — consider self-hosting React if it becomes an issue.
  • If form spam appears: enable hCaptcha in Web3Forms (key is domain-scoped).
